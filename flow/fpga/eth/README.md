# flow/fpga/eth — LiteEth standalone core for karu64 (Phase E0/E1)

Vendored LiteEth MAC for the karu64 VCU118 SoC, generated by `liteeth/gen.py`.
This is the **sim/loopback bring-up** core for `doc/fpga.md` phases
E0–E1: it proves the bus / IRQ / device-tree / Linux-driver path with **no
transceiver work** (a sim MII loopback stands in for the PHY).

## Files

| File | Origin | Notes |
|------|--------|-------|
| `liteeth_sim.yml` | hand-written | gen.py config (source of truth for the build) |
| `regen.sh` | hand-written | reproducible regeneration (pins migen/litex/liteeth) |
| `liteeth_core.v` | **generated** | the MAC core (do not edit; re-run `regen.sh`) |
| `liteeth_csr.csv` / `.json` | **generated** | CSR + memory map (source of truth for the DT) |
| `liteeth_core.xdc` / `.tcl` | **generated** | Vivado project script; `regen.sh` post-processes it so outputs land under `_build` |
| `eth_sim_prims.v` | hand-written | behavioral `FDPE`/`IOBUF` (sim only; Vivado uses real unisims) |
| `eth_mii_loopback.v` | hand-written | sim MII PHY loopback (TX nibbles → RX) |

## Regenerating

```sh
./regen.sh            # uses ~/litex (LITEX_DIR=...) ; bootstraps it if absent
```

Pinned upstream commits (E0 bring-up): migen `e19524c`, litex `61bce27`,
liteeth `456c059`. The generation is `--no-compile` (verilog + CSR export only).

## Configuration

- `core: wishbone` — the LiteX MAC the upstream Linux `liteeth` driver
  (`drivers/net/ethernet/litex/litex_liteeth.c`, compatible `litex,liteeth`)
  binds to. The driver uses **no MDIO/phylib** and calls `netif_carrier_on()`
  unconditionally, so no PHY model is required for the link to come up.
- `phy: LiteEthPHYMII` — a stock, vendor-neutral PHY. In sim its MII pads are
  looped TX→RX by `eth_mii_loopback.v`; on hardware (E3) this PHY is replaced by
  the Xilinx PCS/PMA + DP83867 SGMII front-end.
- `endianness: little`, `csr_data_width: 32`, 2 rx / 2 tx slots, slot size 2048.

## Address map (relocated into karu64's eth MMIO window `0x1100_0000`)

The whole core sits in a single 1 MiB `is_eth()` window (`pa[31:20] == 0x110`),
clear of UART (`0x1000_0000`) and QSPI (`0x1200_0000`):

| Region | Absolute | Size | DT `reg` name |
|--------|----------|------|---------------|
| ctrl CSR | `0x1100_0000` | — | (unused by driver) |
| ethphy CSR (MDIO) | `0x1100_0800` | 0x0a | `mdio` |
| ethmac CSR (control) | `0x1100_1000` | 0x7c | `mac` |
| slot SRAM (rx then tx) | `0x1101_0000` | 0x2000 | `buffer` |

`buffer` holds rx slots first (`0x1101_0000`), tx slots after
(`0x1101_1000` = base + `num_rx_slots*slot_size`), matching the driver.

The wishbone slave is 32-bit data / 30-bit **word** address and decodes the full
absolute address, so the karu64 xbar passes `axi_addr >> 2` to `wishbone_adr`.
karu64's LSU only issues 8-byte-aligned MMIO reads, so the bus bridge returns
both 32-bit halves of the addressed 64-bit word (two wishbone reads) and selects
the write lane from `wstrb` — mirroring the existing PLIC/CLINT MMIO path.

## Device-tree node (goes in `../smoldeb/configs/smoldeb.dts`)

```dts
mac0: ethernet@11001000 {
    compatible = "litex,liteeth";
    reg = <0x11001000 0x7c>,    /* mac    */
          <0x11000800 0x0a>,    /* mdio   */
          <0x11010000 0x2000>;  /* buffer */
    reg-names = "mac", "mdio", "buffer";
    litex,rx-slots = <2>;
    litex,tx-slots = <2>;
    litex,slot-size = <2048>;
    interrupt-parent = <&plic>;
    interrupts = <2>;
};
```

Requires `CONFIG_LITEX=y` + `CONFIG_LITEX_LITEETH=y` in the kernel config and
the PLIC `riscv,ndev` raised to cover source 2.
